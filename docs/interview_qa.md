# 面试话术：费曼学习助手

以下是你向面试官介绍这个项目时可能被问到的问题和推荐回答思路。

---

## Q1: 这个项目是做什么的？技术栈是什么？

**回答思路**：一句话说清楚项目定位，然后列出技术栈，突出后端能力。

> 这是一个基于费曼学习法的 AI 学习助手。用户输入概念，AI 用通俗语言解释并出题测验。后端用 Python Flask，前端是原生 JS。亮点是完整的企业级后端架构：MySQL 持久化、JWT 认证、RESTful API、流式 SSE 响应、Docker 部署、pytest 测试覆盖。

---

## Q2: 为什么选择 Flask 而不是 Django 或 FastAPI？

**回答思路**：这是必问题，展示你对框架选型的理解。

> Flask 轻量灵活，适合 AI 代理类后端——核心逻辑是对接第三方 API 并流式转发，不需要 Django 的 ORM 全家桶。我用 Flask 的**应用工厂模式**来组织代码，这既是 Flask 官方推荐的最佳实践，也能避免循环导入问题。如果换成 FastAPI，异步性能会更好，但 Flask 的生态更成熟，且这个项目的瓶颈在 AI API 的响应速度而非框架开销。

---

## Q3: 讲讲你的数据库设计？为什么只有 3 张表？

**回答思路**：展示有意识的极简设计，而不是"没想清楚"。

> 三张表：users（用户）、conversations（对话分组）、messages（消息内容）。这是**刻意极简**——文件不存表，因为上传后被解析内联到消息正文，解析后的文本就是消息内容。没有 user_settings 表因为当前没有用户可配置的开关。过度归一化和过度设计一样有害，表多了增加 join 开销和代码复杂度。如果后续加功能（比如点赞/收藏），再评估是否需要扩展。

---

## Q4: 你的 JWT 认证是怎么设计的？access token 和 refresh token 有什么区别？

**回答思路**：展示对安全设计的理解。

> access token 有效期 30 分钟，放在 Authorization header 里每次请求携带；refresh token 有效期 7 天，只在 token 过期时用来获取新 access token。这样即使 access token 被泄露，攻击窗口也只有 30 分钟。前端的 api.js 封装了拦截器——任何请求收到 401 时自动用 refresh token 换新 access token 然后重试请求，用户无感知。

---

## Q5: 聊天接口是怎么实现流式输出的？

**回答思路**：展示对 SSE 协议和生成器模式的理解。

> 用的是 Server-Sent Events (SSE)。后端收到用户消息后，调用 Moonshot API 的流式接口，返回一个 Python generator。每读到一行 `data:` 开头的 SSE chunk 就 yield 转发给前端。前端通过 Fetch API 的 ReadableStream 逐块读取并追加到 DOM。选择 SSE 而不是 WebSocket 是因为 AI 流式输出是单向的（服务端→客户端），SSE 基于标准 HTTP，不需要特殊的代理配置，实现更简单。WebSocket 适合双向实时通信场景。

---

## Q6: 为什么要把路由和业务逻辑分到不同层（api/ vs services/）？

**回答思路**：展示对分层架构和可测试性的理解。

> 这是**关注点分离**原则。api/ 层的蓝图函数只处理 HTTP 相关的事——解析请求参数、调用 service、构造响应和状态码。services/ 处理真正的业务逻辑——数据库操作、AI API 调用。这样有两个好处：一是 service 层的函数可以脱离 HTTP 独立测试（不需要模拟请求）；二是便于未来替换实现——比如想把 AI 从 Moonshot 换成 OpenAI，只需要改 service 层一个文件，路由层完全不动。

---

## Q7: 测试策略是怎样的？为什么测试用 SQLite 而不是 MySQL？

**回答思路**：展示务实的工程选择。

> 32 个 pytest 用例覆盖了认证、对话 CRUD、聊天流式、文件解析和健康检查。测试用 SQLite 内存数据库，因为快——没有网络延迟，不需要 Docker，CI 环境也能跑。Moonshot API 调用用 unittest.mock.patch 模拟，不消耗真实 API 额度。我承认这个策略的局限——如果用了 MySQL 特有功能（比如 JSON 字段、全文索引），SQLite 就无法覆盖。但当前项目的 schema 足够简单，SQL 方言差异不影响测试有效性。

---

## Q8: 你做了哪些安全方面的考虑？

**回答思路**：有意识的安全设计比事后补救更好。

> 一是 JWT 双 token 机制限制泄露影响。二是 Flask-Limiter 对聊天接口按 IP 限流（20次/分钟），防止 API 密钥滥用和撞库攻击。三是用户只能访问自己的对话——每次数据库查询都带 user_id 校验，越权返回 403。四是密码用 werkzeug 的 bcrypt 哈希存储，不存明文。五是生产环境用 Gunicorn 替代 Flask 自带 dev server。

---

## Q9: 项目还有哪些可以改进的地方？

**回答思路**：诚实但有思考深度，展示技术广度。

> 短期的话，可以加 Redis 缓存——目前限流是内存模式，重启丢失；多实例部署需要 Redis 统一计数。中期可以加消息队列（Celery）处理文件解析——大文件解析不应阻塞请求。长远看，可以做 API 版本化（/api/v1/），引入 CI/CD（GitHub Actions 自动测试 + 构建镜像推送到 Docker Hub），以及 Kubernetes 编排。但这些对于当前的项目规模来说都是过度设计。

---

## Q10: 这个项目从最开始的原型到现在，你做了哪些改造？

**回答思路**：展示工程化演进能力。

> 最初是 AI 辅助生成的一个原型——单个 app.py + 单个 index.html，没有数据库和认证，纯玩具项目。我做了几件事让它变成能放进简历的项目：
> 1. **架构重构**：单文件拆成 Flask 工厂模式 + Blueprint + Service 分层
> 2. **数据持久化**：加了 MySQL + SQLAlchemy，三表设计
> 3. **用户系统**：JWT 双 token 认证 + 前端登录注册 UI
> 4. **测试体系**：pytest 32 个用例，SQLite 内存库
> 5. **容器化**：Docker + docker-compose 一键启动
> 6. **工程实践**：限流、结构化日志、错误处理、deprecation 警告修复

---

## 面试 Tips

1. **主动说架构决策**：不要等面试官问。介绍项目时说"我特意选了三张表而不是四张，因为..."，这比被动回答更体现思考深度。
2. **承认 trade-off**：比如"SSE 比 WebSocket 简单但只支持单向"，这比只说优点更可信。
3. **量的意识**：知道 32 个测试、20次/分钟限流、30分钟 token 有效期——具体数字让面试官觉得你在认真做。
4. **connect to JD**：如果对方用 Java/Go，可以说"Python 项目是我的学习项目，展示了独立构建后端系统的能力，核心架构思想（分层、认证、测试）是语言无关的。"
